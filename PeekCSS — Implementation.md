# **PeekCSS — Implementation**

## **1\. Project context & goal**

PeekCSS is a Firefox-first (cross-browser via WXT) CSS inspector extension for web developers and designers. It lets users inspect a page's CSS, colors, fonts, and assets without opening DevTools — "hover, see, copy." It's positioned as a maintained, Firefox-native alternative to CSS Peeper, with an eventual Chrome release.

---

## **2\. Locked architecture & constraints**

These are settled decisions. Do not revisit them without an explicit discussion — flag anything that seems to require working around them instead of silently choosing a different approach.

**Stack**

* Framework: WXT (file-based entrypoints, cross-browser build)  
* Language: TypeScript throughout, no exceptions  
* Manifest: MV3

**UI surface**

* The sidebar is the primary UI — `sidebar_action` on Firefox, `sidePanel` on Chrome (these are separate, incompatible APIs; see Cross-browser notes)  
* Popups are ruled out entirely: they close on any page click, which is fatal for an inspector tool. There is no workaround. Only a on-hover popup that follows the cursor is implemented, the main UI stays inside the Sidepanel. 

**Communication**

* Content script ↔ sidebar communicate over a **long-lived port** (`browser.tabs.connect`)  
* One-shot `sendMessage` is *not* used for the hover/click inspection stream — hover/click is a continuous stream, not a transaction  
* `browser.*` namespace is used throughout, never `chrome.*`, for cross-browser compatibility

**Interaction model**

* Click-to-load: hovering highlights an element, clicking loads its data into the sidebar  
* Hover-to-Show: Hovering shows a popup at the mouse cursor with data overview  
* Both functionalities are toggle-able, not always active

**Persistent state**

* All persistent state goes through `browser.storage.local` — never held only in background/service-worker memory

**Authored CSS values**

* Where possible, retrieve original authored values by walking `document.styleSheets` / `cssRules`, not just `getComputedStyle()` (which returns resolved values only)  
* Known caveats: cross-origin stylesheet access restrictions, CSS-in-JS (runtime-injected styles not always traceable to a stylesheet rule), cascade resolution complexity (specificity, `!important`, layers), and inaccessible UA stylesheet rules  
* Treat this as best-effort with graceful fallback to computed values — not a guarantee, and not a blocker for features that depend on it

## **3\. File / folder map**

This project follows WXT's standard layout. The sidebar app is plain TypeScript + DOM (no UI framework). `components/` exists but is currently empty. Builds land in `.output/{firefox,chrome}-mv3[-dev]/`.

```
entrypoints/
├── background.ts          MV3 background: toolbar click → sidebarAction.toggle();
│                          relays http(s) download requests to the downloads API
├── content.ts             Content script — ALL overlay code lives here:
│                          createHighlight() (hover highlight), createPopup()
│                          (cursor-following hover popup), read() (element
│                          inspection), scanOverview()/scanImages(); owns the
│                          content-script side of the long-lived port
└── sidepanel/             THE SIDEBAR APP (Firefox sidebar_action / Chrome
    │                      side_panel, generated by WXT from this entrypoint)
    ├── index.html         Markup for #view-inspector, #view-overview,
    │                      #view-settings + bottom nav
    ├── main.ts            Orchestrator: tabs.connect port lifecycle, inspector/
    │                      popup toggles, shortcuts, color-format pref, download
    │                      routing
    ├── inspectorView.ts   Inspector tab rendering
    ├── overviewView.ts    Overview tab rendering (palette, images, a11y report)
    ├── navigation.ts      Bottom-nav controller (inspector/overview/settings)
    ├── theme.ts           Light/dark theme (persists to storage.local)
    └── style.css

utils/
├── messages.ts            PORT-MESSAGE TYPES: INSPECTOR_PORT name,
│                          InspectorMessage (content → sidebar), SidepanelMessage
│                          (sidebar → content), DownloadRequest/DownloadResult
│                          (sidebar ↔ background one-shot), plus InspectionData,
│                          OverviewData, AccessibilityReport, ImageInfo
├── color.ts               Color parsing/formatting, contrast math, ColorFormat
├── audit.ts               Page accessibility audit (contrast/text-size/color-blind)
├── clipboard.ts           Copy helper
└── download.ts            data-URL → Blob, filename/extension helpers
```

**Storage helpers: none exist yet.** `browser.storage.local` is called inline in two places (`sidepanel/theme.ts` → `theme` key; `sidepanel/main.ts` → `colorFormat` key). New code should introduce:

* `utils/storage.ts` — typed get/set helpers for settings keys (migrate the two inline call sites)  
* `utils/tier.ts` — `getUserTier()`, cached account/license state, dev override (§7)

## **4\. Existing features (baseline — already implemented)**

* Four sub-tabs: **Inspector**, **Overview**, **Typography**, **Settings**  
* **Inspector** *(anonymous)*  
  * Point-and-click to inspect  
  * Loads relevant CSS for the selected element  
  * Shows spacing and contrast (numeric ratio \+ AA/AAA pass/fail)  
  * Copy whole CSS, selected groups, or single values — with inline "Copied ✓" feedback  
  * Empty-container hover resolves down to the nearest visible child  
* **Overview** *(free account)*  
  * Contrast / web-accessibility score  
  * Lists potential contrast issues  
  * Color palette with per-purpose tallies; "By purpose" grouping *(Pro)*  
  * Palette export: CSS variables, SCSS, JSON, Tailwind *(Pro)*  
  * Shows all images on the page, with instant download; "Download all" as ZIP with sequential fallback *(Pro)*  
* **Typography** *(free account)*  
  * All fonts used on the page, with weights, sizes, and usage counts  
  * Sort by usage or by role (heading / body / UI heuristic)  
* **Settings** *(free account)*  
  * Light/dark mode toggle  
  * Preferred color format  
  * Font unit preference (px / rem / pt), applied app-wide  
  * Keyboard shortcut reference  
  * Dev builds: tier override selector  
* **Tier gating** (§7) — `utils/tier.ts` \+ `gating.ts`; locked views/features shown disabled with lock icon \+ tier hint; default tier `anonymous`; dev override via `devTierOverride` storage key

Status of §5/§6: all items shipped except **font pairing suggestions** (blocked on §9: API choice \+ privacy copy) and the items already moved to §9 (detachable sidebar, dock position). PDF/ASE palette export remains a stretch goal.

## **5\. Planned features**

### **Overview tab**

1. **Sort color palette by purpose** — group extracted colors by where they're used (background, text, border, etc.), based on the CSS properties they came from. This is a rule-based grouping, not the AI-driven "semantic palette naming" feature under separate consideration — keep these as distinct features even if they end up adjacent in the UI.  
2. **"Download all" for images** — bundle as a ZIP (preferred) or fall back to sequential individual downloads via the `downloads` API. Note: ZIP requires bundling a library (e.g. JSZip), which works against the "keep the bundle lean" principle — worth a quick size check before committing, but not a blocker.  
3. **Export color palette in multiple formats** — start with text-based formats that are cheap to generate: CSS custom properties, SCSS variables, JSON, Tailwind config object. Treat PDF and Adobe ASE as stretch goals — PDF generation in particular pulls in a non-trivial library for what may be a rarely-used export path; consider whether that's worth the bundle cost vs. a web-based export tool later.

### **New tab: Typography**

1. Show all fonts used on the page  
2. Sort by role (heading, body, UI, etc.) — heuristic based on element tags/contexts (h1–h6, p, button, etc.), no AI needed  
3. Suggest similar free fonts via an external API (Google Fonts / Fontspring)  
   * This sends detected font names to a third-party service — even though it's not page content, this is a third-party data flow and should be disclosed in the store listing per the project's existing privacy principles for external API calls  
   * Gating tier for this tab is part of the open decision in §7 — don't hardcode it yet

## **7\. Account, free tier & Pro**

PeekCSS uses three tiers: **Free (no account)**, **Free (with account)**, and **Pro**.

* **Free (no account)** covers the core single-element inspection workflow — the "hover, see, copy" experience that has to work the moment someone installs the extension, with zero signup friction.  
* **Free (with account)** adds page-level analysis (Overview, Typography) and customization (Settings) — a real upgrade in capability, at no cost to the user or to us, in exchange for an email address.  
* **Pro** adds bulk operations, exports, source-level depth, and anything with an ongoing per-use cost (third-party APIs, storage/history).

### **Gating interface implication**

The above requires a tier check with three states, not a boolean:

type UserTier \= 'anonymous' | 'free\_account' | 'pro';

function getUserTier(): UserTier { /\* reads cached account/license state \*/ }

Every gated feature checks against this rather than a simple `isPro()` boolean. For development, this should be easily overridable (storage key or env flag) to test all three tiers without a real backend.

## **9\. Open questions / needs research before implementation**

* **Detachable / floating sidebar**: Firefox's `sidebarAction` API has no detach-to-floating-window capability — the sidebar is a docked browser pane, full stop. An alternative would be a *separate* extension surface (a standalone window opened via `browser.windows.create`, loading the same Vue app) with its own port connection and lifecycle — this is a new surface to design, not a setting on the existing one. Worth discussing whether it's worth the added complexity before scoping it.  
* **Dock position setting**: which side the sidebar docks (left/right) is a browser-level user preference in Firefox, not exposed to extensions via the sidebar APIs. Chrome's sidePanel API is explicitly not compatible with sidebarAction either, so this can't be a cross-browser settings toggle as currently scoped. At most, Settings could link to the relevant browser instructions — confirm whether that's worth including.  
* **Account/Pro model** (§7) — needs an explicit decision before any gating code is written.  
* **Typography tab gating tier** — depends on §7's outcome.  
* **Third-party font API choice** (Google Fonts vs. Fontspring) and the associated privacy disclosure copy for the store listing.  
* **New manifest permissions** (e.g. `downloads` for the ZIP export) — flag any new permission for review, since it affects store listing review and the permissions prompt users see on install.

## **10\. Verification & process**

**Build/typecheck (Fable should run these before considering a task done)**

* `tsc --noEmit` clean  
* `wxt build` succeeds for both Firefox and Chrome targets

**Manual QA checklist (per feature, for Leon to run)**

* Load unpacked build in Firefox, confirm sidebar opens and the feature behaves as specced  
* Repeat for Chrome where the feature is cross-browser

**Git / commit process**

* Commit after completing each coherent task; commit between tasks, not just at the end of a session  
* Clear, short commit messages  
* After implementing or materially changing a feature, update §4 (Existing features) to reflect the new state — this doc should stay accurate as a snapshot of what's actually built
